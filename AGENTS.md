# 编码规范

## 命名约定

- **类名、成员函数**：大驼峰命名法（PascalCase）`。
- **类成员变量（私有）**：以 `m_` 开头，小驼峰命名法。
- **成员函数**：使用动词或动宾短语描述功能，例如：`CalculateTotal`、`GetUserInfo`。
- **常量名**：全大写，单词间用下划线分隔，例如：`MY_CONSTANT`。
- **全局函数**：全小写，单词以'_'分割。
- **全局变量**：以 `g_` 开头，小驼峰命名法。
- **单字符名称**：避免使用单字符作为名称，除非是临时变量或循环计数器（如 `i`、`j`、`k`）。

## 格式规范

- **缩进**：使用四个空格进行缩进，不使用 Tab。
- **操作符空格**：二元操作符前后添加空格，如 `a = b + c;`。
- **代码布局**：使用适当的空格和换行提高可读性，保持代码布局一致。
- **代码格式**：函数和关键字的括号都单独放一行。
- **行长限制**：每行代码控制在 120 个字符之内。
- **逗号后空格**：逗号后添加一个空格，如 `int a, b, c;`。
- **函数调用/定义**：函数名和括号之间不留空格，如 `void functionName()`。
- **控制语句**：关键字后加一个空格再加括号，如 `if (condition) { ... }`左括号与关键字同行。

## 注释规范

- **语言**：统一使用中文注释。
- **成员变量**：必须添加注释，说明其作用。
- **成员函数**：必须添加注释，说明函数作用、输入参数和返回值。
- **头文件**：需有文件级注释，说明模块用途。
- **代码变更**：代码变更时必须同步更新对应注释，保持注释与代码一致。
- **复杂逻辑**：状态机、协议交互等复杂逻辑需添加 `Why` 说明，而不仅是 `What`。

## 测试验证

- 每次开发完代码必须测试验证，包括问题场景模拟和边界场景。
- 代码编译通过是最低验证要求。
- 新增/修改功能需覆盖正常路径和异常分支。
- 涉及协议交互的场景需模拟对端行为。

## 提交要求
本项目所有 git commit message 必须遵循 Conventional Commits 规范。
提交标题格式必须为：
<type>(<scope>): <summary>
常用 type：
- feat：新增功能
- fix：修复问题
- refactor：重构
- docs：文档
- test：测试
- chore：杂项
- perf：性能优化
- build：构建相关
- revert：回滚到上一个版本
- ci：CI/CD 相关
提交要求：
- summary 必须简洁明确，直接说明本次改了什么
- 不要使用“修改代码”“优化一下”“调整逻辑”“修复一些问题”这类模糊描述
- 提交标题必须保持 Conventional Commits 兼容，不要在最前面增加 [Codex]、[ai] 等前缀
如本次提交由 AI 工具参与生成、修改或辅助完成，必须在 commit message 正文中增加一行：
AI: Codex
示例：
fix: 修复工单状态回写异常
AI: Codex
feat: 新增设备离线告警聚合逻辑
AI: Codex
refactor: 重构报文解析流程
AI: Codex
